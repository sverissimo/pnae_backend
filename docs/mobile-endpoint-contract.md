# Mobile → Backend endpoint contract (reference)

Quick reference of every backend HTTP endpoint the **frozen** mobile app
(`../../pnae_mobile/`) calls, for vetting backend changes against the
mobile-compatibility hard rule. Mobile is shipped on thousands of devices and
cannot be updated, so the request/response behaviour below is a contract.

Generated by auditing `pnae_mobile/` (not only `@infrastructure/api/` — also
`@services/`, `@sync/`, `features/`, `@domain/`). Re-verify with a full-tree grep
before trusting it; mobile call sites live outside the API folder too.

## How mobile carries auth

Every request sends `Authorization: Bearer <CLIENT_TOKEN>` (static shared
secret). `AuthMiddleware` short-circuits and leaves `req.user` **undefined**.
Mobile-facing endpoints must not rely on `req.user` for identity — the payload
carries `tecnicoId`/`produtorId` explicitly.

## How mobile reads responses (`@infrastructure/api/API.ts`)

- **GET** → `response.json()` (parsed object/array).
- **POST/PATCH** → `response.text()` (plain string; callers `JSON.parse` when needed).
  This is why `PlainTextExceptionFilter` flattens errors to plain text.

## Endpoints mobile does NOT call (safe to change/remove)

- `GET /atendimento/getTemasAtendimento` — **removed**; mobile never called it.
- `GET /atendimento/getReplacedAtendimentos` — web hot-path only.
- `GET /relatorios/dashboard`, `PATCH /atendimento/:atendimentoId/aprovar`,
  `PATCH /atendimento/:atendimentoId/pendencia`,
  `PATCH /atendimento/:atendimentoId/aprovar-sei`,
  `PATCH /atendimento/:atendimentoId/remover-aprovacao-sei` — web-only additions
  (all keyed on `atendimentoId` alone; the two SEI routes are admin-only; see
  backend AGENTS.md).
- `POST /atendimento/findMany`, `/relatorios/zip/*` — not used by mobile.

## SQLite gotcha (relatório) — the load-bearing constraint

Mobile stores relatórios in **SQLite with a fixed schema**
(`@infrastructure/database/relatorio/dto/RelatorioLocalDTO.ts`). On write it runs
`humps.decamelizeKeys` over **all** fields and the generic DAO builds the column
list from the object keys.

- `SQL_DAO.create` filters out `null`/`undefined` keys → an all-null unknown
  field is harmless; a **non-null** unknown field becomes a column in
  `INSERT INTO relatorio (...)` → SQLite **"no such column"**.
- `createMany` uses `Promise.allSettled` → such a failure is swallowed (the row
  silently fails to persist), not thrown.
- `SQL_DAO.update` does **not** filter nulls — but mobile only `update`s with
  local data or sync-stripped data, so it is not exposed in practice.

**Therefore any new RelatorioModel field returned to mobile must be stripped on
the backend on EVERY mobile-facing relatório read path, not just sync.** The
canonical whitelist is `MOBILE_RELATORIO_FIELDS` with the `toMobileRelatorio()`
projection helper (`src/modules/@sync/utils/mobile-relatorio-fields.ts`). It is
applied on both mobile read paths:

- sync — `SyncService` maps `outdatedOnClient`/`missingOnClient` through it.
- `GET /relatorios?produtorId=` — the `findByProdutorId` controller route maps
  its result through it (the route is mobile-only; web never queries by
  produtorId, so the strip is unconditional).

This is what keeps the `comercializaPnae`/`produtoTratado`/`grauInteresse`
columns (written by the web form) from reaching mobile's SQLite.

The whitelist must list **only** what mobile can store — being a valid backend
`RelatorioModel` key is not enough. In particular `atendimentoAnteriorId` is a
backend-derived field set during atendimento-id replacement
(`RelatorioDomainService.updateAtendimentoIds`); it has no mobile column, so it
must stay **out** of the whitelist even though it is a legitimate `RelatorioModel`
key. The replacement target `atendimentoId` (mapped to `id_at_atendimento`) is
kept; only the predecessor id is dropped.

Produtor uses AsyncStorage (JSON), so extra produtor fields are tolerated.
