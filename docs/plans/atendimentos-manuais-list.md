# Plan — Atendimentos com relatório manual: list + combined PDF

Two web-only features over Demeter atendimentos whose relatório was **not** produced by the PNAE app:

1. **List** them (paginated metadata table) — sourced from a new upstream gql query.
2. **Build the generated manual PDF** (perfil header + proof-of-visit image files) — the current
   **Ver relatório** action.

Plus `GET /atendimento/getArquivos` for individual raw-file downloads. Current frontend actions are:
open the generated manual PDF, download the raw relatório PDF, and download the raw proof-of-visit
photo. All additive; mobile is untouched.

For the later plan that copies the raw relatório PDF pages into the generated manual PDF, see
`docs/plans/atendimentos-manuais-file-download-options.md`.

## Domain context (read this first — the naming only makes sense with it)

- Every tecnico action is an **atendimento** in the external **Demeter** DB (via `gql_server` + the
  legacy REST API), not PNAE's Postgres.
- Atendimentos created **by the PNAE app** and atendimentos created **elsewhere** are *identical rows*
  in Demeter. The **only** discriminator is **`link_pdf`** — the URL of the app-generated relatório
  PDF. `link_pdf IS NULL` ⇒ the relatório for this atendimento was **not** generated by the app.
- "Elsewhere" = a legacy Demeter web interface (manual data entry) **or** a genuinely hand-written
  relatório. So the atendimento itself is *not* "manual" — what's manual/non-app is the **relatório**.
- A **manual relatório may have multiple files** stored in Demeter, fetched via the existing
  `getArquivos` endpoint by `fileType`:
  - `fileType=relatorio` → the printed relatório, **signed by hand by the produtor**. For valid
    PNAE rows this is `application/pdf`; the frontend can download it separately as the raw relatório
    file.
  - `fileType=foto` → **proof-of-visit photo** (tecnico + produtor together at the property).
  PNAE keeps these semantic `fileType` values at its public route boundary, but the REST call to the
  Demeter gateway maps them to the stored `tipo_arquivo` MIME families (`relatorio` →
  `application/pdf`; `foto` → `image/jpeg`, `image/png`, `image/gif`). DOCX/TIFF are absent for PNAE,
  and the few `application/msword` rows found in broader DB checks are treated as bad data.
- Upstream filters the list server-side to `link_pdf IS NULL` **AND** indicador `4550` (Demeter's own
  "relatório manual" tag), narrowing ~2M atendimentos to **~100k (≈5%)**. The `4550` literal and the
  filter live **upstream** — we never replicate them.

> **Pin this definition in code (comment at the DTO/route), because no identifier can carry it:**
> *"manual relatório = relatório not generated by the PNAE app (`link_pdf IS NULL` + indicador 4550);
> the atendimento row is identical to app-created ones. The files are `relatorio` = signed printed PDF,
> and `foto` = proof-of-visit image."*

## Naming

The thing we list is an **atendimento that *has* a manual relatório** — the `com` keeps `atendimento`
as the entity and attaches "manual" to the relatório, where it belongs.

| Concern | Name |
| --- | --- |
| List route | `GET /atendimento/com-relatorio-manual` |
| List service | `listAtendimentosComRelatorioManual({ pageSize, cursor })` |
| Output DTO | `AtendimentoComRelatorioManualDTO` / `AtendimentoComRelatorioManualPageDTO` |
| gql client + query | `getAtendimentosComRelatorioManual` / `atendimentosComRelatorioManualQuery` — **mirrors the literal upstream wire field** `atendimentosComRelatorioManual` (don't rename; it's the field we call) |
| data-mapper | `AtendimentoDataMapper.toComRelatorioManual(item, localidade)` / `toComRelatorioManualPage(page, localidadeMap)` |
| Individual file fetch | `GET /atendimento/getArquivos?atendimentoId=&fileType=foto\|relatorio` — raw PDF/photo downloads |
| Combined PDF route | `GET /relatorios/manual?id=<base64urlToken>` — intentionally public/no-auth; `id` is an obfuscated public PDF token, not an auth token |
| PDF input builder | `RelatorioExportService.createManualPdfInput(atendimentoId)` |
| PDF generator | `PdfGenerator.generateManualPdf(...)` + template `relatorio-manual.ejs` |

Why the slightly-long `com-relatorio-manual` over the terser provenance name `atendimentosForaDoApp`
(also honest — these atendimentos *were* created outside the app): **cohesion**. The file endpoint and
the PDF endpoint both revolve around the "relatório manual" concept; tying the list to the same noun
keeps the three endpoints obviously one feature. `foraDoApp` would read as a disconnected provenance
filter. It's a one-token swap if you prefer it — localized to this section.

(`OutsidePnaeApp` is out — code identifiers stay pt-BR per AGENTS.)

## Upstream contract (gql_server, reviewed & aligned)

We consume **only the paginated list query** (no single-item query — the "detail" is the PDF/file
download). The manual filter is baked in server-side; we send `pageSize`/`cursor` plus the authenticated
web user's trusted scope (`id_usuario`, `id_reg_empresa`) derived from `req.user`.

```graphql
type AtendimentoListItem {
  id_at_atendimento: BigInt!
  data_inicio_atendimento: DateTime!   data_fim_atendimento: DateTime
  data_validacao: DateTime             data_atualizacao: DateTime!   data_criacao: DateTime!
  data_sei: DateTime                   data_see: DateTime
  sn_pendencia: Int                    sn_validado: Int
  dt_update_record: DateTime           id_at_anterior: BigInt
  id_und_empresa: String!              ativo: Boolean!
  clientes: [AtendimentoClienteResumo!]!   # produtor + propriedade (N, but PNAE uses [0])
  usuarios: [AtendimentoUsuarioResumo!]!   # tecnico(s) (N, but PNAE uses [0])
  arquivos: [ArquivoRelatorioManualResumo!]! # raw file metadata; binary is not selected here
}
type ArquivoRelatorioManualResumo { idArquivo: BigInt! tipoArquivo: String! nomeArquivo: String! }
type AtendimentoPage { items: [AtendimentoListItem!]!  pageSize: Int!  nextCursor: BigInt  hasMore: Boolean! }
# upstream field (confirmed):
# atendimentosComRelatorioManual(
#   pageSize: Int = 200
#   cursor: BigInt
#   id_usuario: BigInt
#   id_reg_empresa: String
# ): AtendimentoPage!
```

- **Cursor/keyset over the PK**, `id_at_atendimento DESC` (creation order, not business-date — upstream
  can't index `data_inicio`). We pass `cursor` opaquely and echo `nextCursor`/`hasMore`.
- **BigInt scalars arrive as strings**, and the inbound `cursor` is accepted as a string by the
  upstream `BigInt` scalar → no coercion / `BigIntInterceptor` work on our side.
- **Scope is pushed upstream before pagination.** Admin/developer sends no scope; coordenador regional
  sends `id_usuario + id_reg_empresa`; staff sends only `id_usuario`. The gateway is service-token
  protected and trusts this backend-provided scope.
- Sparse staff scopes are the upstream performance case to verify: without a Demeter index led by
  `at_atendimento_usuario.id_usuario`, a tecnico with few/old manual relatórios can make the gateway
  scan much farther back through `at_atendimento` to fill one page. Rollout needs upstream
  `EXPLAIN (ANALYZE, BUFFERS)` for a sparse `id_usuario`, first and deep page.
- Upstream returns `clientes`/`usuarios` in **stable child-id order**, so our `[0]` pick is
  deterministic.
- Upstream returns `arquivos[]` ordered by lowest `idArquivo` first. Each item exposes raw
  `tipoArquivo` plus app-generated `nomeArquivo`; the file binary is not selected in the list query.
- Upstream **delegates município/regional enrichment to us** (Phase 1, localidade cache).

# Phase 1 — List endpoint (`atendimentos com relatório manual`)

End-to-end paginated list: new gql query → flatten `[0]` → localidade enrichment → controller route.
Self-contained; depends only on the upstream `atendimentosComRelatorioManual` field being live (or mocked).

### 1a. gql query + client
- `src/@graphQL-server/queries/atendimento-queries.ts` — add `atendimentosComRelatorioManualQuery`, selecting the
  `at_atendimento` scalars **plus** `clientes { produtor { nm_pessoa nr_cpf_cnpj dap caf } propriedade
  { nome_propriedade geo_ponto_texto } }` and `usuarios { id_usuario nome_usuario id_und_empresa }`.
  Variables: `pageSize: Int`, `cursor: BigInt`, `id_usuario: BigInt`, `id_reg_empresa: String`.
  **Don't over-select** — requesting produtor/propriedade is what makes upstream run its one expensive
  `at_cli_atend_prop` scan per page; request only what the table shows.
- `src/@graphQL-server/atendimento-api.service.ts` — add `getAtendimentosComRelatorioManual({ pageSize, cursor })`
  returning the upstream `AtendimentoPage`.

### 1b. DTO file
- `src/modules/atendimento/dto/atendimento-com-relatorio-manual.dto.ts` — input query type
  (`{ pageSize?: number; cursor?: string; id_usuario?: string; id_reg_empresa?: string | null }`),
  upstream item/page types, and the output
  `AtendimentoComRelatorioManualDTO` / `...PageDTO`. Shared across controller/service/data-mapper, so
  it lives in `dto/` (per AGENTS "no hidden reusable types"). **Put the domain-definition comment here.**

Output item shape (adjustable — FE comes later):

```ts
type AtendimentoComRelatorioManualDTO = {
  // scalars passed through from the upstream atendimento item
  id_at_atendimento: string;
  data_inicio_atendimento: string;   data_fim_atendimento: string | null;
  data_validacao: string | null;     data_atualizacao: string;   data_criacao: string;
  data_sei: string | null;           data_see: string | null;
  sn_pendencia: number | null;       sn_validado: number | null;
  dt_update_record: string | null;   id_at_anterior: string | null;
  id_und_empresa: string;            ativo: boolean;

  // arrays collapsed to first element (nested, faithful to upstream sub-shapes)
  produtor: { nm_pessoa; nr_cpf_cnpj; dap; caf } | null;
  propriedade: { nome_propriedade; geo_ponto_texto } | null;
  usuario: { id_usuario; nome_usuario; id_und_empresa } | null;

  // enrichment from getMunicipiosEmater (Redis-cached)
  nomeMunicipio: string | null;   id_reg_empresa: string | null;   nomeRegional: string | null;

  // Demeter file metadata for table actions
  arquivos: { idArquivo: string; tipoArquivo: string; nomeArquivo: string }[];
};
type AtendimentoComRelatorioManualPageDTO = {
  items: AtendimentoComRelatorioManualDTO[];
  pageSize: number;   nextCursor: string | null;   hasMore: boolean;
};
```

### 1c. data-mapper (the only place shaping happens) — KISS
`src/modules/atendimento/data-mapper/atendimento.data-mapper.ts` — add `toComRelatorioManual(item,
localidade)` and `toComRelatorioManualPage(page, localidadeMap)`:
- `clientes[0].produtor` → `produtor`, `clientes[0].propriedade` → `propriedade`; `usuarios[0]` →
  `usuario` (first only — PNAE guarantees one; extras ignored).
- Do **not** resolve produtor→propriedades through `pl_propriedade_ger_pessoa` in this feature. The
  upstream `clientes[]` array already carries the atendimento-specific produtor/propriedade pair from
  `at_cli_atend_prop`; if an edge case has more than one pair, v1 keeps the KISS rule and uses
  `clientes[0]`.
- Empty arrays → `null` (don't throw).
- Localidade keyed on the atendimento-level `id_und_empresa` (e.g. `"H1234"`): add `nomeMunicipio`,
  `id_reg_empresa`, `nomeRegional`. Unmapped unit (or a `"G…"` regional id) → `null` fields, no error.
- Page envelope (`items`/`pageSize`/`nextCursor`/`hasMore`) kept identical to upstream so the cursor
  contract passes straight through. The mapper is pure — the service feeds it the localidade map.

### 1d. localidade cache — extend `CachedMunicipiosReader`
This is the consumer half upstream deliberately left to us (keeps near-static names off every page row,
avoids duplicating `H%`/`G%` logic in two services). `getMunicipiosEmater` is already Redis-cached, but
the reader currently stores only `[id_und_empresa, regional_id]` and exposes `getUnidadeToRegionalMap()`.
We need three fields, all present on `MunicipioEmater` (`nome_municipio`, `regional_id`, `nome_regional`).

**Recommended (DRY):** refactor to cache the full projection once and derive two views —
- private `loadAll()` caches the full `MunicipioEmater[]` (hundreds of rows), keeps the in-flight dedup;
- `getUnidadeToRegionalMap()` derives the existing `Map<unit, regional_id>` (**behaviour unchanged**);
- **new** `getUnidadeToLocalidadeMap()` → `Map<unit, { nomeMunicipio; id_reg_empresa; nomeRegional }>`;
- bump `CACHE_KEYS.municipiosEmater` `:v1 → :v2` (blob shape changes; old key self-heals on expiry).

⚠️ `getUnidadeToRegionalMap()` feeds `getAtendimentoAuthScope` (**auth path**) — its behaviour and spec
must stay green. *Lower-risk fallback:* add `getUnidadeToLocalidadeMap()` under a separate cache key
and leave the existing method fully untouched (costs a second cold-cache fetch of the same endpoint).
Caveat: `getMunicipiosEmater` covers only active `H%` units → null-tolerate misses (the mapper already does).

### 1e. service + controller
- `src/modules/atendimento/atendimento.service.ts` — `listAtendimentosComRelatorioManual({ pageSize,
  cursor })`: call `getAtendimentosComRelatorioManual` + `getUnidadeToLocalidadeMap()`, hand both to
  `toComRelatorioManualPage`, return it.
- `src/modules/atendimento/atendimento.controller.ts` — `GET /atendimento/com-relatorio-manual`,
  **declared above `@Get(':id')`** (single static segment, else shadowed — same reason `/getArquivos`
  sits above `:id`). Validate inline (mirroring `getArquivos`, no global ValidationPipe): coerce
  `pageSize` to number, default **200**, clamp `[1, 1000]` (same as upstream); `cursor`
  optional string passthrough. Derive upstream scope from authenticated `req.user`: admin/developer
  sends no scope, coordenador regional sends `id_usuario + id_reg_empresa`, staff sends only
  `id_usuario`; missing/unsupported users get 403.

**No module wiring changes** — `AtendimentoGraphQLAPI` and `CachedMunicipiosReader` are already
providers in `AtendimentoModule` and injected into `AtendimentoService`.

### 1f. tests + docs
- `atendimento.data-mapper.spec.ts` — flatten/first-element, empty arrays → null, enrichment hit/miss.
- Service spec — pageSize default 200, cursor passthrough, envelope passthrough.
- Update the cached-municipios reader spec (regional map unchanged + new localidade map).
- AGENTS "Endpoints not used by mobile" — add `GET /atendimento/com-relatorio-manual`.

# Phase 2 — Generated "relatório manual" PDF

A PDF that reuses the existing perfil header, then renders the manual relatório's available
proof-of-visit images instead of the produtos/relatório-data block. Keyed on **`atendimentoId`** and served from
`relatorios.controller` (the link conceptually loads a *relatório*, not an atendimento). Independent of
Phase 1 (it uses `getArquivos` + produtor/perfil, not the list), so it can be built in parallel.

Current frontend Phase 1 also provides separate raw downloads for the relatório PDF and proof photo via
`GET /atendimento/getArquivos`. Copying the raw relatório PDF pages into this generated PDF is deferred
to `docs/plans/atendimentos-manuais-file-download-options.md`.

### 2a. PDF input builder
`src/modules/relatorios/relatorios.export.service.ts` — add `createManualPdfInput(atendimentoId)`
(the service already injects `atendimentoService`, `produtorApi`, `produtorService`):
- Resolve the produtor from the atendimento: `atendimentoService.findOne(atendimentoId)` →
  `at_cli_atend_prop.id_pessoa_demeter` (+ `at_atendimento_usuario.id_usuario`, `id_und_empresa`).
- `produtorApi.getProdutorById(produtorId)` → `perfis`/`propriedades`; build `perfilPDFModel` reusing
  the **exact** perfil logic from `createPDFInput` (`new Perfil().toPDFModel({ ...perfil, nome_propriedade })`).
- Fetch image files via the existing path: `atendimentoService.getArquivos({ atendimentoId,
  fileType: 'foto' })` → yields a `{ buffer, contentType }` (decodeArquivo already sniffs image bytes;
  the REST client maps this file type to Demeter image MIME `tipo_arquivo` values before calling the
  gateway). Convert to
  `data:${contentType};base64,${buffer.toString('base64')}`.
- Return `{ perfilPDFModel, produtor: { nomeProdutor, cpf, caf, id_und_empresa }, atendimento meta,
  imagens: [{ dataUri, legenda: 'Comprovação de visita' }] }`.

**Robustness (KISS):**
- **Missing perfil** — manual atendimentos may have no perfil. Don't throw like `createPDFInput`;
  fall back to a minimal cover (produtor + atendimento metadata you already have) + the images.
- **Missing image** — render the cover only; never fail the whole PDF because no image file is present.
- **Don't hardcode `image/jpeg`** — use the sniffed `contentType` in the data URI (the current
  relatório template hardcodes jpeg; our pics may be png).

### 2b. template + generator
- `src/@pdf-gen/templates/relatorio-manual.ejs` — `<%- include('perfil', …) %>` (reuse the partial),
  then one full-width `<img src="<dataUri>">` per picture, `page-break` between them, with a caption.
  Reuses `header.ejs`/`footer.ejs` and `relatorioStyles.ejs` `.imageContainer/.foto` classes.
- `src/@pdf-gen/pdf-gen.ts` — add `generateManualPdf(input)`: a slimmer sibling of `generatePdf` that
  renders `relatorio-manual.ejs` and pipes through the **same** `wkhtmltopdf(...)` config + header/footer
  writing. Feeds base64 directly (skip `Base64ImageService`/`FileResolver`, which read local disk — we
  already have the bytes from `getArquivos`).

### 2c. route
`src/modules/relatorios/relatorios.controller.ts` — add public/no-auth
`GET /relatorios/manual?id=<base64urlToken>`, mirroring `generatePdf` (set `Content-Type:
application/pdf`, `Content-Disposition: inline; filename=…`, `pdfStream.pipe(res)`), decoding the
query token to `atendimentoId`, then calling `relatorioExportService.createManualPdfInput` and
`PdfGenerator.generateManualPdf`.

> **Public route decision.** This route intentionally does **not** rely on the `auth_token` cookie or
> any frontend auth handling. Backend authorization/filtering happens at the list-data layer before
> the user receives rows. The PDF token is only friction/obfuscation for regular users, not a security
> guard: it prevents obvious browser-bar guessing like changing `/987` to `/988`, but anyone who
> understands the code can reverse it.
>
> **Token contract:** frontend builds `value = <5 random decimal digits> + atendimentoId`, encodes it
> as base64url (`+`→`-`, `/`→`_`, strip `=` padding), and opens
> `${BASE_URL}/relatorios/manual?id=${token}`. Backend decodes base64url to UTF-8, validates the first
> five chars are digits, slices them off, validates the remaining `atendimentoId` is decimal digits,
> then loads the PDF. Call it an **obfuscated public PDF token**, never an auth token.

### 2d. tests + docs
- Spec `createManualPdfInput` — produtor/perfil resolution from `atendimentoId`, the **missing-perfil**
  and **missing-picture** fallbacks, correct data-URI contentType.
- Controller spec — `GET /relatorios/manual?id=<base64urlToken>` decodes the token, strips the 5-digit
  noise prefix, rejects malformed tokens with 400, and streams the PDF.
- PDF generation itself is hard to unit-test (wkhtmltopdf is a binary); cover template rendering
  (`ejs.renderFile`) and input assembly, and smoke-test the route manually.
- AGENTS "Endpoints not used by mobile" — add `GET /relatorios/manual?id=<base64urlToken>`, noting it
  loads a manual relatório (perfil + proof-of-visit image), is public/no-auth, and uses a 5-digit-noise
  base64url token as obfuscation only.

## Cross-cutting

### Mobile compatibility — additive only ✅
- No existing controller method, route, request/response shape, status code, or error format changes.
- New paths (`/atendimento/com-relatorio-manual`, `/relatorios/manual?id=<base64urlToken>`) are brand-new;
  mobile cannot call them. The only ordering change (the list route above `@Get(':id')`) is internal
  to the controller and alters no path mobile uses.
- `getArquivos` keeps the same public route shape, but now probes only the PNAE-supported MIME set and
  is used by the frontend for separate raw relatório/photo downloads.
- The new upstream gql field is **distinct** from `atendimentos(ids)`/`atendimento(id)` so existing
  flows keep working.

### Out of scope / deferred
- Extra user-driven filters/sorts (date ranges, producer name, arbitrary unit filters). They need
  upstream indexes/DBA review before rollout. The v1 authorization scope is already pushed upstream
  via `id_usuario` / `id_reg_empresa` before pagination.
- Live DB validation of the upstream sparse-user query plan. This backend sends the right trusted
  scope, but only Demeter `EXPLAIN (ANALYZE, BUFFERS)` can close the performance risk for rare staff
  matches.
- **Page-level caching** of the list (cursor pages cache poorly); only the municipios cache is reused.
- **Renaming `getArquivos`** to a relatório-manual-specific name — deferred per request (kept for
  testing). The generic name is fine for now.
- **OCR** (picture → text → stored relatório) — future brainstorm only; handwriting needs a cloud OCR
  model and carries an LGPD/PII decision. Not in this plan.
- Future generated-PDF assembly that copies the raw relatório PDF pages into `/relatorios/manual`.
